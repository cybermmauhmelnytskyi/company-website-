## 📝 Про проєкт
[cite_start]Цей проєкт є повною симуляцією реальної командної розробки у веб-студії, виконаною в індивідуальному форматі[cite: 24, 27]. [cite_start]Головна мета — комплексна перевірка навичок роботи з Git за методологією Git Flow, керування завданнями через Issues, вирішення конфліктів та налаштування деплою[cite: 25].

[cite_start]В межах спринту було створено легкий, адаптивний та інформативний **"Довідника команд Git"** у вигляді вебсайту[cite: 27]. [cite_start]Усі етапи командної взаємодії (розподіл тасків, робота у функціональних гілках, Code Review та злиття конфліктів) були змодельовані та успішно реалізовані одним автором[cite: 25, 28].

## 🔄 Симуляція командного процесу (Workflow)
[cite_start]Для виконання критеріїв практичної роботи було чітко відтворено життєвий цикл розробки[cite: 28]:
1. [cite_start]**Захист гілки `main`:** Було увімкнено Branch Protection, що забороняє прямі пуші, змушуючи розробника вносити зміни виключно через Pull Requests[cite: 31, 40].
2. [cite_start]**Розгалуження (Feature Branches):** Під кожне з трьох Issues було створено окрему гілку (наприклад, `feature-local`, `feature-branches` тощо)[cite: 32, 37]. [cite_start]Код писався ізольовано[cite: 38].
3. [cite_start]**Зв'язування з тасками:** Коміти оформлювалися із ключовими словами для автоматичного закриття завдань (наприклад, `Closes #1`, `Closes #2`)[cite: 39].
4. [cite_start]**Code Review та Конфлікти:** Було проведено перехресне рецензування та коментування власних Pull Requests для демонстрації навичок перевірки коду[cite: 41, 42]. [cite_start]Конфлікти злиття (Merge Conflicts), які виникли через одночасне редагування файлу `index.html`, були успішно вирішені вручну[cite: 43].
