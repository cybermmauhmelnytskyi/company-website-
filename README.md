## 📝 Про проєкт
Цей проєкт є повною симуляцією реальної командної розробки у вебстудії, виконаною в індивідуальному форматі. Головна мета — комплексна перевірка навичок роботи з Git за методологією Git Flow, керування завданнями через Issues, вирішення конфліктів та налаштування деплою.

В межах спринту було створено легкий, адаптивний та інформативний **"Довідник команд Git"** у вигляді вебсайту. Усі етапи командної взаємодії (розподіл тасків, робота у функціональних гілках, Code Review та злиття конфліктів) були змодельовані та успішно реалізовані одним автором.

---

## 👤 Виконавець проєкту
Оскільки проєкт виконувався індивідуально, усі передбачені сценарієм ролі були успішно змодельовані та реалізовані одним розробником:

* **Розробник / Лідер:** Хмельницький Нікіта

### 🗺️ Моделювання ролей та зон відповідальності:
1. **Лідер (Капітан)** — ініціалізував репозиторій, розгорнув базовий скелет `index.html`, налаштував правила захисту головної гілки (**Branch Protection**) та створив завдання у вкладці Issues.
2. **Розробник 1 (Локальна робота)** — реалізував блок документації, присвячений базовим локальним командам Git (Issue #1).
3. **Розробник 2 (Робота з гілками)** — реалізував блок документації для команд розгалуження та злиття (Issue #2).
4. **Розробник 3 (Робота з GitHub)** — описав команди для взаємодії з віддаленим сервером (Issue #3).

## 🔄 Симуляція командного процесу (Workflow)
Для виконання критеріїв практичної роботи було чітко відтворено життєвий цикл розробки:
1. **Захист гілки `main`:** було увімкнено Branch Protection, що забороняє прямі пуші, змушуючи розробника вносити зміни виключно через Pull Requests.
2. **Розгалуження (Feature Branches):** під кожне з трьох Issues було створено окрему гілку (наприклад, `feature-local`, `feature-branches` тощо). Код писався ізольовано.
3. **Зв'язування з тасками:** коміти оформлювалися із ключовими словами для автоматичного закриття завдань (наприклад, `Closes #1`, `Closes #2`).
4. **Code Review та Конфлікти:** було проведено перехресне рецензування та коментування власних Pull Requests для демонстрації навичок перевірки коду. Конфлікти злиття (Merge Conflicts), які виникли через одночасне редагування файлу `index.html`, були успішно вирішені вручну.
